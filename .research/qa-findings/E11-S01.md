# QA Findings — E11-S01

## Flagged for a follow-up story

- `transformGeoRows` hardcodes `fiscalYear: 2024` on every produced snapshot.
  `fetchGeoSnapshots` passes `params.fiscalYear` only into the API
  `time_period` filter, but never overrides the stamped `fiscalYear` on the
  returned `GeoSpendingSnapshot` objects (unlike `fetchSpendingByAgency`,
  which does override it). Result: snapshots are always stamped `fiscalYear: 2024`
  regardless of the requested year. `sync.service.ts#syncGeography` then upserts
  by the `{ stateCode, fiscalYear, agencyId, scope }` unique key using
  `snapshot.fiscalYear`, so any non-FY2024 sync is written into the FY2024 bucket
  (silent data-integrity corruption). Not fixed in QA because correcting it is an
  observable behavior change (returned `fiscalYear` values would change) and the
  existing spec has no row pinning non-2024 fiscal years. Fix: thread
  `fiscalYear` (and likely `agencyId`) into `transformGeoRows` instead of the
  magic `2024` / `0` defaults, and add table rows for a non-2024 year.

- `fetchSpendingByAgency` builds the `/search/spending_by_geography/` body with
  no `filters.agencies`, so the API returns nationwide spending across ALL
  agencies, yet every returned row is stamped with a single
  `agencyId = parseInt(params.agency, 10)` (which also collapses a toptier code
  like `'080'` to the integer `80`). The method's name and signature imply
  per-agency attribution that the request body does not actually request. Needs
  confirmation against the intended sync semantics in `sync.service.ts` (out of
  scope for this QA closed set) before changing — likely either a
  `filters.agencies` clause should be added or the method's contract/return type
  should be reworked. Not fixed in QA: doing so is a behavior change and a
  public-API consideration (story notes forbid changing service return types).
