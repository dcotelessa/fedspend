# QA Findings — E15-S03

## Flagged for a follow-up story

- **`agency.id` is `undefined` when spending rows are upserted in `syncAgenciesAndSpending` (latent data-integrity bug):** `UsaSpendingService.fetchAgencies()` builds agencies with `id: undefined as any`. `syncAgenciesAndSpending` then calls `agencyRepo.upsert(agency, ['toptierCode'])`, which returns an `InsertResult` and does NOT populate the generated id back onto the in-memory `agency` object. `agenciesList` is sliced from those same id-less objects, so downstream `spendingRepo.delete({ agencyId: agency.id, fiscalYear: year })` runs with `agencyId: undefined` (TypeORM ignores `undefined` in a where clause → deletes ALL rows for that fiscalYear across every agency), and `spendingRepo.upsert({ ...record, agencyId: agency.id }, [...])` upserts with `agencyId: undefined` against a NOT NULL column. The table-driven `sync.service.spec.ts` masks this because every mock `agencyFetchResult` supplies agencies that already have `id` set (e.g. `{ id: 7, ... }`). This is pre-existing — it predates the E15-S03 award-type work and is orthogonal to it — and the fix (capture the upsert return / re-query agencies by `toptierCode` before spending sync) is a behavior change that belongs in its own story with its own red-green coverage of the id-population path.

- **`time_period` fiscal-year window looks off-by-one vs. the US federal FY convention:** every spending/geo/disaster fetch builds `time_period: [{ start_date: '${year}-10-01', end_date: '${year+1}-09-30' }]`. Federal FY `N` actually runs Oct 1 of `N-1` → Sep 30 of `N`, so a row stored as `fiscalYear: 2024` is sourced from the date window USASpending itself would label FY 2025. The behavior is identical across `fetchSpendingByAgency`, `fetchGeoSnapshots`, and `fetchDisasterSpending`, and is locked in by the table-driven specs (e.g. `start_date: '2024-10-01'` for `fiscalYear: 2024`), so it may be a deliberate project convention. QA cannot change it: doing so is an observable behavior change and the correct window needs verification against the live USASpending API (the same "verify-first" risk called out in the E14-S08/S10 `awarding_agencies` episode). A follow-up story should confirm the intended labeling against live data before any date adjustment.

- **`FetchSpendingResult.total` is semantically meaningless and unused:** `fetchSpendingByAgency` accumulates `total += rawGeo.length`, i.e. a count of state-level geography rows across award types — not an award count and not a dollar sum. No consumer reads it (`sync.service.ts` only iterates `spendingResult.rows`). Removing the field would change the method's public return type, which is outside a behavior-preserving QA pass; a future story should either drop it or repurpose it to an actual total (e.g. summed obligated cents) once a consumer needs it.
