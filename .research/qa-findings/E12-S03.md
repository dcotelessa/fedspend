# QA Findings — E12-S03

## Flagged for a follow-up story
- **Migrate component styles to theme CSS variables instead of hardcoded light fallbacks.**
  `dashboard.component.scss` declares `background: var(--card-bg, #fff)` and
  `color: var(--primary, #1976d2)` with light-mode fallbacks, but the theme never
  defines `--card-bg`/`--primary`, so dark-mode correctness currently depends on
  `styles.scss` overriding the result with `!important` rules of *lower* specificity.
  QA patched the card case by defining `--card-bg` on `.dark-theme`, but the same
  latent pattern exists for any future component that sets a background/color at
  higher specificity than the global `.dark-theme <tag> { !important }` override.
  Proper fix: introduce a shared set of design-token custom properties
  (`--fs-bg`, `--fs-surface`, `--fs-text`, `--fs-primary`, ...) defined for both
  light and dark on the theme scope, and have component SCSS consume them with no
  hardcoded fallback. This touches multiple component `.scss` files (outside this
  story's two-file closed set) and is behavior-preserving only if done project-wide.
- **Frontend bundle exceeds the configured budget (566 kB of 500 kB).**
  Pre-existing, unrelated to theme work, and not affected by this QA pass. Needs a
  separate code-splitting / lazy-loading story. Flagged only so it is not lost.

## Resolved in this QA pass (behavior-preserving, no follow-up needed)
- Extracted the inline dark/light palette (`#e0e0e0`, `#1e1e1e`, `#333`, ...) into
  named SCSS variables so the palette has a single source of truth. Compiled CSS
  is byte-identical in color values.
- Defined `--card-bg` on `.dark-theme` so the dashboard nav cards resolve to the
  dark surface through the component's intended variable hook rather than relying
  on a cross-selector `!important` cascade win.
