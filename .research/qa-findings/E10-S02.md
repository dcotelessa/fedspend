# QA Findings — E10-S02
## Flagged for a follow-up story

- Toggle does not live-apply the theme to the DOM: `AppComponent.ngOnInit`
  reads `theme.isDark$()` once and calls `document.body.classList.toggle('dark-theme', isDark)`
  a single time. The `(change)="theme.toggle()"` handler updates the signal and
  localStorage but nothing re-applies the body class, so clicking the
  mat-slide-toggle moves the switch (the `[checked]` is signal-bound) yet the
  page only re-themes after a full reload. Fix needs a reactive `effect()` that
  re-runs `classList.toggle` when `isDark$` changes — an observable-behavior
  change, so it cannot be made in a behavior-preserving QA pass.

- Dual, misaligned theming mechanisms in styles.scss: the
  `@media (prefers-color-scheme: dark)` block targets `:root:not([data-theme="light"])`,
  but `data-theme` is never written anywhere — the JS path applies
  `body.dark-theme` instead. Because the attribute is never `"light"`, the
  selector always matches, so a user whose OS is in dark mode gets a dark
  `:root` background (and inherited `#e0e0e0` text color) bleeding through even
  when the app's explicit theme is light. Fix requires collapsing to one
  mechanism (e.g., write `data-theme` on `document.documentElement` from the
  service/effect and key every rule off it) — a behavior change, out of scope.

- Chart.js text color does not react to the theme: the story notes require
  charts to look sharp on dark backgrounds via `Chart.defaults.color` driven by
  `ThemeService`, but no `Chart.defaults` / `defaults.color` wiring exists
  anywhere in `frontend/src`. The chart components live outside this story's
  closed file set, so this needs its own story (and its own spec) rather than a
  QA edit.
