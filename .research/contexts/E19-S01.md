## Notes
Deep-architecture cleanup (see AGENTS.md Skill 1 + Skill 3). SpendingModule is a shallow phantom: spending.module.ts is `export class SpendingModule {}` with an empty controller and service, while SpendingRecord is actually read via AgenciesService (@InjectRepository(SpendingRecord)). Dissolve the stubs; KEEP backend/src/spending/spending-record.entity.ts and backend/src/spending/dto/ (still used). DTO pattern: each read controller takes @Query() dto: XQueryDto (class-validator decorators @IsInt/@IsString/@IsOptional/@Min/@Max) instead of individual @Query string params. CRITICAL DETAIL: with transform:true the pipe coerces, but query params arrive as STRINGS — number fields in the DTO need @Type(() => Number) from class-transformer to coerce before @IsInt validates. Mirror the existing geography QueryStatesInput / disaster QueryStatesParams field shapes. GeographyController currently relies on implicit number coercion of @Query; DisasterController currently does manual parseInt(fiscalYear,10) — both become a single validated DTO. AgencySpotlightQueryDto already exists in backend/src/agencies/dto/ but is vestigial (not wired) — either wire it or add a new AgencyListQueryDto for the list endpoint and delete the vestigial one. Tests are table-driven (AGENTS.md Skill 2): for each DTO, rows of valid input (passes through) + bad input (non-int, out-of-range, unknown field) rejected by the pipe. Controller specs mock the service and assert the DTO is forwarded as the typed object. Do NOT touch the Sync module or any write pipeline — those are already exemplary deep modules.
