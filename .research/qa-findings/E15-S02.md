# QA Findings — E15-S02

## Flagged for a follow-up story

- [Controller `fiscalYear` parsing propagates `NaN` for non-numeric input]:
  `AgenciesController.list` does `fiscalYear ? Number(fiscalYear) : undefined`.
  For a non-numeric query string (e.g. `?fiscalYear=abc`), `Number('abc')` is
  `NaN`, which is then passed to `AgenciesService.findAllWithTotals`. Because
  `NaN` is non-nullish, the `fiscalYear ?? currentFiscalYear()` fallback is
  skipped and the JOIN runs with `sr.fiscalYear = NaN`, matching no rows (every
  agency returns `totalCents: 0`). The sane behavior is to fall back to the
  default fiscal year (Skill 1 boundary enforcement, e.g.
  `const fy = Number(fiscalYear); ... Number.isFinite(fy) ? fy : undefined`).
  Not fixed here because changing `NaN` → default is an observable behavior
  change beyond a behavior-preserving QA pass. Note: this input is not yet sent
  by the frontend (the param is opt-in for a future FY-selector story), so the
  blast radius today is nil, but it should be hardened before that selector
  ships.

- [`syncAgenciesAndSpending` deletes per-year spending before fetching, risking
  data loss on a failed fetch]: The per-(agencyId, fiscalYear) loop calls
  `spendingRepo.delete({ agencyId, fiscalYear })` BEFORE
  `usaService.fetchSpendingByAgency`. If that fetch fails or returns a non-
  `success` status for one year, that year's existing rows are already wiped and
  nothing replaces them. The sibling `syncGeography` uses the safer order
  (fetch first, then delete+save only on `success`), so a transient USA Spending
  outage preserves existing geo data but erases existing spending data. With the
  new multi-year loop (5 years x up to 20 agencies = 100 fetches) the failure
  surface is larger than before. Aligning agencies sync to fetch-first (matching
  geography) is a behavior change (data preservation on failure), so it is left
  for a follow-up rather than introduced here.
