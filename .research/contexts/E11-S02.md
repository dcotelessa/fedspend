## Notes
Three bugs prevent the sync from running end-to-end. E11-S01 fixed the API URLs and response parsing. This story fixes the remaining data-layer bugs.

BUG 1: recovery-ratio.ts returns Infinity (SQLite cannot store it)
  File: recovery-ratio.ts
  Current: if (femaCents === 0) return Infinity;
  Error: SqliteError: no such column: Infinity
  Fix: return 0 when femaCents === 0 (no FEMA funding = no ratio to compute).
  Also guard: if either input is NaN, return 0.
  Update recovery-ratio.spec.ts: the test row for femaCents=0 fedCents>0 should expect 0, not Infinity.

BUG 2: upsertBy causes UNIQUE constraint failures
  File: sync.service.ts
  Current: findOne(match) -> save(entity). When id is undefined, TypeORM tries INSERT. If the record already exists (race, re-sync, stale data), INSERT fails with UNIQUE constraint.
  Error: SqliteError: UNIQUE constraint failed: agency.toptierCode
  Fix: replace upsertBy with TypeORM's repo.upsert() method:
    await repo.upsert(entity, conflictColumns);
  For agencies: conflict on ['toptierCode']
  For spending records: conflict on ['agencyId','fiscalYear','quarter','awardTypeLabel']
  For geo snapshots: conflict on ['stateCode','fiscalYear','agencyId','scope']
  For disaster records: conflict on ['defGroup','stateCode']
  Note: SQLite supports ON CONFLICT DO UPDATE. TypeORM's upsert() uses this internally.
  If repo.upsert() is unavailable in this TypeORM version, use createQueryBuilder()
    .insert().into(Entity).values(entity)
    .orUpdate(['columnName'], ['conflictColumn'])
    .execute();

BUG 3: transformGeoRows hardcodes fiscalYear
  File: usa-spending.service.ts
  Current: transformGeoRows sets fiscalYear: 0 on every snapshot.
  Fix: pass fiscalYear as a third parameter:
    transformGeoRows(rows, scope, fiscalYear)
  Callers: fetchSpendingByAgency knows params.fiscalYear.
           fetchGeoSnapshots knows params.fiscalYear.
           fetchDisasterSpending uses FY2024 (hardcoded in the POST body).
