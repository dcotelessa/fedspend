# QA Findings — E12-S02

## Flagged for a follow-up story

- [Per-state records instead of one-per-agency sum]: Story notes require
  "Create one SpendingRecord per agency per fiscal year with
  awardTypeLabel='Total'", summing all states' `aggregated_amount` into a
  single agency total. Current `fetchSpendingByAgency` emits one
  SpendingRecord per state (rows = transformed geo rows). In
  `sync.service.ts`, those rows are upserted with conflict target
  `[agencyId, fiscalYear, quarter, awardTypeLabel]`, so every state row for a
  given agency collides and only the LAST upserted state survives — the
  stored `obligatedAmount` is the last state's amount, not the agency sum.
  Fixing it changes observable DB writes (behavior change), so it is out of
  scope for behavior-preserving QA. Needs a dedicated story: aggregate per
  agency (sum cents) and emit exactly one SpendingRecord per agency per FY.

- [`total` field semantics / dead data]: `FetchSpendingResult.total` is set
  to `rawGeo.length` (a count of state rows), but the story intent was the
  summed agency total. No consumer reads `.total` (verified: zero references
  in `sync.service.ts` / `sync.service.spec.ts`). Either compute the real
  sum or drop the field from the return type. Left unchanged here because
  altering the returned value/type is a behavior + public-API change.

- [Unused `agency` param in `fetchGeoSnapshots`]: The `agency` field of the
  `params` object is destructured into the signature but never used in the
  POST body — geo snapshots are fetched agency-agnostic. It is dead, but it
  is part of the public method signature exercised by the spec
  (`fetchGeoSnapshots({ agency: '080', ... })`). Removing it would change
  the public API, so it is flagged rather than removed in QA.
